## Context

FFBayes currently has three different dashboard touchpoints that can drift apart: the canonical runtime artifacts generated by `ffbayes draft-strategy`, the cheap HTML regeneration path exposed by `ffbayes refresh-dashboard`, and the repo-tracked `site/` bundle that `ffbayes publish-pages` deploys to GitHub Pages. The repo already proved that a stale runtime HTML file can survive long enough to be restaged into `site/`, so the real problem is lifecycle hygiene rather than model correctness.

The current automation footprint is split across local CLI commands, runtime shortcut copies, Pages staging, and a GitHub Pages deployment workflow that only knows how to publish `site/`. That means source-of-truth boundaries are implicit today and stale artifacts are only obvious when a smoke test or manual check catches them.

Stakeholders are the local draft operator, who needs a cheap repair path during iterative dashboard work, and the repo maintainer, who needs a reliable way to keep runtime, site, and template changes aligned without rerunning the full draft-strategy pipeline every time.

## Goals / Non-Goals

**Goals:**
- Make dashboard regeneration cheap when the payload is already available.
- Make stale runtime or staged dashboard artifacts detectable before they get published or committed.
- Clarify which command owns which artifact family: `draft-strategy` for canonical payload generation, `refresh-dashboard` for HTML regeneration, and `publish-pages` for Pages staging.
- Keep current filenames, runtime roots, and `site/` targets stable while improving lifecycle hygiene.
- Add automation that checks or enforces sync between source templates, generated dashboard HTML, and the committed Pages bundle.

**Non-Goals:**
- Reworking the draft model, backtest methodology, or freshness semantics from the analysis layer.
- Introducing live draft provider sync or board-state ingestion.
- Renaming the canonical dashboard files or moving `site/` out of the repo-tracked Pages path.
- Replacing `path_constants.py` or the existing CLI dispatch model.

## Decisions

### 1. Treat the runtime payload as the canonical source for cheap dashboard regeneration

The lightweight repair path should regenerate HTML from an existing dashboard payload rather than rerunning the model pipeline. That keeps the expensive draft-strategy computation separate from the cheap rendering and staging path.

Why this over rerunning `draft-strategy`:
- The operator often needs only a template refresh, not a new model pass.
- It avoids unnecessary recomputation when the payload is already current.

Alternatives considered:
- Rebuild everything on each refresh. Rejected because it is too expensive for routine dashboard hygiene.
- Hand-edit the staged `site/` HTML. Rejected because it breaks the source-of-truth boundary and invites drift.

### 2. Detect drift by regeneration comparison rather than timestamps

The lifecycle check should compare the staged dashboard against what the current payload and template would generate, rather than trusting mtimes or copy timestamps.

Why this over timestamp checks:
- Timestamps are fragile across copies, platforms, and publish flows.
- Regeneration comparison directly answers the user question: “does this artifact still match the current source?”

Alternatives considered:
- Embed a separate version stamp and rely on it alone. Rejected because the source template can change without a stamp update, which would miss exactly the drift we care about.

### 3. Keep `publish-pages` as the staging boundary, but make it a verified staging boundary

`publish-pages` should remain the command that copies runtime dashboard artifacts into `site/`, but it should be paired with a drift check so stale site output is not silently promoted.

Why this over a new publishing system:
- It preserves the current Pages contract and artifact paths.
- It keeps source-of-truth boundaries simple: runtime artifacts are generated first, then staged into `site/`.

Alternatives considered:
- Generate `site/` directly in the Pages workflow without a local staging step. Rejected because the repo already uses local runtime artifacts as the canonical pre-publish source, and the staging step is useful for inspection and smoke testing.

### 4. Validate `site/` in CI using the same regeneration path used locally

The GitHub Actions workflow should verify that the committed `site/` bundle is still in sync with the current dashboard generator before allowing stale changes to linger. The check should use the same refresh path available to local operators so the CI behavior mirrors the manual repair path.

Why this over a bespoke CI-only checker:
- It avoids maintaining two separate notions of “fresh enough.”
- It gives maintainers a direct command to run locally when CI fails.

Alternatives considered:
- Let the Pages workflow deploy blindly and rely on manual smoke tests. Rejected because it already allowed drift to persist.

### 5. Keep dashboard sync validation separate from the Pages deploy workflow

Dashboard sync validation should live in a dedicated GitHub Actions workflow that runs on pull requests and on pushes touching dashboard/template artifacts, while `pages.yml` remains responsible only for deploying the already-validated `site/` bundle.

Why this over embedding validation inside `pages.yml`:
- The existing Pages workflow is deployment-oriented and only triggers when `site/` changes on `master`, which is too late for useful drift detection.
- A separate validation workflow surfaces stale artifacts before merge instead of discovering them only at deploy time.
- It keeps deploy responsibilities simple: validate first, deploy second.

Alternatives considered:
- Add sync validation directly to `pages.yml`. Rejected because it mixes pre-merge correctness checks with post-merge deployment and would still fail too late in common drift scenarios.

### 6. Make `refresh-dashboard` emit an explicit machine-readable freshness result

The refresh path should expose a structured stale/fresh result so CI and local operators can reason about lifecycle state without parsing ad hoc console text. The result should be derived from regeneration comparison rather than from timestamps alone.

Why this over shell-level diff logic alone:
- CI needs a stable contract for pass/fail decisions.
- Local operators benefit from explicit reporting about what was checked, what drifted, and which paths are authoritative.
- A structured result makes future automation easier without changing the source-of-truth rule that regeneration comparison is decisive.

Alternatives considered:
- Rely purely on stdout text and shell diffs. Rejected because it is brittle and harder to evolve or test.

### 7. Keep one regeneration primitive and add a check mode instead of a new top-level command

The repo should continue to use `refresh-dashboard` as the single regeneration primitive, extended with a non-mutating check mode such as `--check --json`, rather than introducing a separate `dashboard-lifecycle-check` command immediately.

Why this over adding a dedicated check command:
- It avoids CLI sprawl while the lifecycle surface is still small.
- It reinforces the mental model that regeneration and validation are two modes of the same dashboard lifecycle primitive.
- It keeps the local operator workflow simple: generate, check, and optionally stage from one command family.

Alternatives considered:
- Introduce a distinct lifecycle-check command now. Rejected because it creates more surface area before the current refresh path has matured.

## Risks / Trade-offs

- [Regeneration-based checks add a bit of runtime cost] → Keep the check scoped to the dashboard payload and HTML regeneration path, which is much cheaper than rerunning the model pipeline.
- [CI can only validate the repo-tracked `site/` bundle, not a developer’s private runtime tree] → Document the local refresh path clearly and make stale `site/` drift fail fast in CI.
- [Operators may confuse `refresh-dashboard` with `draft-strategy`] → Keep the command names and README guidance explicit about source-of-truth ownership.
- [Workflow changes can surprise existing release habits] → Preserve `site/index.html` and `site/dashboard_payload.json` targets, and add a local smoke path so the workflow mirrors real operator behavior.

## Migration Plan

1. Keep the existing `draft-strategy` output layout and `site/` staging targets stable.
2. Use `refresh-dashboard` as the cheap regeneration and repair path for runtime dashboard HTML, and extend it with a machine-readable check mode.
3. Add a dedicated dashboard-sync validation workflow that compares committed `site/` output against regeneration from the tracked payload before merge.
4. Keep `pages.yml` focused on deploying validated `site/` output rather than acting as the first place drift is detected.
5. Update tests and README guidance so operators know when to use `draft-strategy`, `refresh-dashboard`, `refresh-dashboard --check`, and `publish-pages`.
6. If the validation path proves too strict in practice, temporarily relax the check to warning-only while keeping the local refresh command intact.
